## 一、项目概述

实现一个单列 AI 广告推荐信息流 App。核心功能：上下滑动浏览广告卡片、点赞/收藏/评论持久化、AI 对话搜索推荐、视频播放、曝光/点击埋点统计。

### 1.1 技术栈

| 层级 | 技术 | 为什么选它 |
|------|------|-----------|
| 语言 | Java 11 | 零基础学习的第一门语言 |
| UI 框架 | XML 布局 + ViewBinding | 可视化拖拽 + 代码绑定，易理解 |
| 列表 | RecyclerView (多 ViewType) | 高性能列表，支持多种卡片样式 |
| 架构 | MVVM (ViewModel + LiveData) | Google 官方推荐，数据驱动 UI |
| 数据库 | Room (SQLite 封装) | 编译时 SQL 检查，自动生成代码 |
| 网络 | Retrofit2 + OkHttp | 类型安全的 HTTP 客户端 |
| 视频 | ExoPlayer (media3) | Google 官方视频播放器 |
| AI | DeepSeek API / Mock 兜底 | OpenAI 兼容格式，换模型只需改 URL |
| JSON | Gson | Google 出品，轻量级解析 |

## 三、设计核心：可复用模块与面向接口

### 3.0 什么是"复用"？（零基础解释）

> **复用 = 一个组件写一次，在多处使用**

比如你做了一个"点赞按钮"，主页要用，详情页也要用。如果写两遍——改一个地方就要改两个地方，容易漏。复用就是写成一个组件，两处都引用它——改一次，两处都更新。

### 3.1 InteractionBar：三个按钮的组合复用

```java
// InteractionBar 是一个自定义 View（LinearLayout 的子类）
public class InteractionBar extends LinearLayout {
    // 包含: 点赞图标+数字 | 收藏图标+数字 | 评论图标+数字
}
```

**复用关系**：

| 使用位置 | 怎么引用的 | 文件 |
|---------|-----------|------|
| 主界面广告卡片 | `view_ad_card.xml` 中直接写 `<InteractionBar>` | `view_ad_card.xml` |
| 小图卡片 | `item_ad_small_image.xml` 中也写 `<InteractionBar>` | `item_ad_small_image.xml` |
| 详情页 | `activity_detail.xml` 中写 `<InteractionBar>` | `activity_detail.xml` |

```xml
<!-- 三个布局里完全一样的写法 -->
<com.tao.android.ai_ad_recommendation.ui.component.InteractionBar
    android:id="@+id/card_interaction_bar"
    android:layout_width="match_parent"
    android:layout_height="wrap_content" />
```

**组件对外接口（API）**：

```java
// 设置点赞状态（数据驱动UI，组件自己不存数据！）
interactionBar.setLiked(true, 5);   // 红心 + 显示 5
interactionBar.setLiked(false, 0);  // 空心 + 显示 0

// 设置收藏状态
interactionBar.setFavorited(true, 3);

// 设置评论数
interactionBar.setCommentCount(10);

// 设置点击回调
interactionBar.setOnInteractionListener(new OnInteractionListener() {
    void onLikeClick();       // 用户点了点赞
    void onFavoriteClick();   // 用户点了收藏
    void onCommentClick();    // 用户点了评论
});
```

### 3.2 VideoPlayerView：ExoPlayer 的封装复用

```java
// VideoPlayerView 封装了 ExoPlayer 的全部复杂操作
public class VideoPlayerView extends FrameLayout {
    // 对外暴露简单接口，内部管理 ExoPlayer 生命周期
}
```

**对外接口**：

| 方法 | 作用 | 哪里用 |
|------|------|--------|
| `setupVideo(url)` | 初始化 + 加载视频 | DetailActivity |
| `play()` / `pause()` | 播放/暂停 | DetailActivity |
| `setMuted(true/false)` | 静音切换 | DetailActivity (false=有声) |
| `showController()` | 显示控制栏(进度条+全屏按钮) | DetailActivity |
| `releasePlayer()` | 释放资源 | onDetachedFromWindow 自动触发 |

**为什么要封装？**

| 不封装 | 封装后 |
|--------|--------|
| 每个用播放器的地方都要写 20 行 ExoPlayer 初始化代码 | 调用 `setupVideo(url)` 一行搞定 |
| 换播放器引擎要改所有文件 | 只改 VideoPlayerView 内部 |
| 内存泄漏风险分散 | 集中在 `onDetachedFromWindow` 一处管理 |

### 3.3 分层解耦的 AI 服务设计

```
MockAiService          AiApiService (Retrofit)
(本地关键词匹配)        (DeepSeek API)
     ↘                    ↙
    AiChatHelper (统一入口，含自动降级)
```

```java
// AiChatHelper 是统一入口，内部自动选择：
// - 有 API Key → 调大模型
// - 无 API Key / 失败 → MockAiService 关键词匹配兜底
public class AiChatHelper {
    private final AiApiService apiService;   // Retrofit 接口
    private final MockAiService fallback;    // 本地兜底

    public void analyzeAndMatch(String query, List<String> history,
                                List<AdItem> allAds, MatchCallback callback) {
        if (noApiKey) {
            fallback.semanticSearch(query);  // 本地关键词
        } else {
            apiService.chat(...).enqueue();  // 大模型
        }
    }
}
```

**降级机制**：大模型不可用时（无 API Key、网络异常、返回格式错误），自动回退到 `MockAiService.semanticSearch()`，保证 app 在任何情况下都能跑。调用方（SearchViewModel）不需要感知内部如何选择。这也是"面向接口/面向抽象"思想的落地——不过这里接口是隐式的：`AiChatHelper` 就是契约，MockAiService 和 AiApiService 都是实现细节。

### 3.4 Repository 模式

```
ViewModel (不关心数据从哪来)
    ↓ 调用
Repository (协调员，决定用哪个数据源)
    ↙        ↘
Room(本地)   Mock/API(远程)
```

```java

public List<AdItem> loadInitialAds() {
    mockDataSource.loadMockData();  // 现在用 Mock
```

## 四、核心技术详解

### 4.1 Room 数据库（替代手写 SQLite）

**什么是 Room**：Google 官方的 SQLite 封装库，用注解自动生成数据库操作代码。

**传统 SQLite 的问题**：
```java
// 手写SQL，编译不检查，运行时错了才知道
db.execSQL("INSERT INTO user_behaviors (ad_id, behavior_type) VALUES ('123', 'like')");
Cursor c = db.rawQuery("SELECT * FROM ... WHERE ad_id = ?", new String[]{"123"});
// 拿到 Cursor 后还要手写循环转成 Java 对象
```

**Room 的解决方案**：
```java
// 1. 定义一个接口，写 SQL 注解
@Dao
public interface BehaviorDao {
    @Insert
    void insertBehavior(UserBehavior behavior);  // Room 自动生成 INSERT

    @Query("SELECT COUNT(*) FROM user_behaviors WHERE ad_id = :adId AND behavior_type = :type")
    LiveData<Integer> getBehaviorCount(String adId, String type);  // 返回 LiveData, 数据变了自动通知
}

// 2. 调用时只关心业务
behaviorDao.insertBehavior(new UserBehavior(adId, "like", null, System.currentTimeMillis()));
```

| 对比点 | 手写 SQLite | Room |
|--------|-----------|------|
| SQL 语法检查 | 运行时崩溃 | 编译时就报错 |
| Java 对象映射 | 手动 Cursor 转 | 自动 |
| 数据变化通知 | 自己写回调 | LiveData 自动推 |
| 代码量 | 多 | 少 (注解生成) |

**三个核心注解**：

| 注解 | 位置 | 作用 |
|------|------|------|
| `@Entity` | 类上 | 标记这个类是数据库表 |
| `@Dao` | 接口上 | 标记这个接口是数据操作对象 |
| `@Database` | 抽象类上 | 告诉 Room 有哪些表、哪个版本 |

### 4.2 LiveData（会主动通知的数据容器）

**什么是 LiveData**：一个"活的数据"，值变了会自动通知所有观察者。

```java
// 普通变量：改了 UI 不会自己刷新
int count = 0;
count = 5;
textView.setText(String.valueOf(count)); // 每次都要手动刷新

// LiveData：改了自动刷新
MutableLiveData<Integer> count = new MutableLiveData<>(0);

// 注册观察者（只写一次）
count.observe(this, value -> {
    textView.setText(String.valueOf(value)); // 值变了自动调
});

// 改值
count.setValue(5);  // → 自动触发 observe 里那段代码
```

**和生命周期的关系**：LiveData 知道 Activity/Fragment 是否还活着。页面销毁了会自动取消观察，不会内存泄漏。

### 4.3 Retrofit（HTTP 网络请求）

**什么是 Retrofit**：把 HTTP API 定义成 Java 接口，自动处理网络请求。

```java
// 1. 定义接口
public interface AiApiService {
    @POST
    Call<Map<String, Object>> chat(
        @Url String url,                   // 请求地址
        @Header("Authorization") String auth, // 请求头
        @Body Map<String, Object> body        // 请求体
    );
}

// 2. 创建实例
Retrofit retrofit = new Retrofit.Builder()
    .baseUrl("https://api.example.com/")
    .addConverterFactory(GsonConverterFactory.create()) // JSON 自动转 Java 对象
    .build();

AiApiService api = retrofit.create(AiApiService.class);

// 3. 发请求
api.chat(url, "Bearer xxx", body).enqueue(callback); // enqueue = 异步，不卡主线程
```

**关键点**：`enqueue` 在后台线程发请求，不会卡 UI。和 BehaviorRepository 里 `executor.execute()` 是同样的线程思想。

### 4.4 RecyclerView（高性能列表）

**RecyclerView 三部曲**：

```
1. LayoutManager  →  决定怎么排列（竖向/横向/网格）
2. Adapter        →  决定显示什么数据、长什么样
3. ViewHolder     →  缓存 item 视图，避免重复创建
```

**多 ViewType（本项目的核心用法）**：

```java
@Override
public int getItemViewType(int position) {
    switch (adList.get(position).getType()) {
        case "large_image": return 0;  // 大图卡片
        case "small_image": return 1;  // 小图卡片
        case "video":       return 2;  // 视频卡片
    }
    return 0;
}

@Override
public ViewHolder onCreateViewHolder(ViewGroup parent, int viewType) {
    if (viewType == 1) {
        // 加载小图布局
        return new ViewHolder(inflater.inflate(R.layout.item_ad_small_image, parent, false));
    } else {
        // 加载大图/视频布局
        return new ViewHolder(inflater.inflate(R.layout.view_ad_card, parent, false));
    }
}
```

**为什么高效**：ViewHolder 模式缓存已创建的 View。屏幕上有 5 个 item，RecyclerView 只创建约 7 个 ViewHolder，滚动时复用而不是新建。

## 五、核心功能实现原理

### 5.1 数据流：MVVM 架构

```
用户操作(滑动/点击) → Fragment/Activity
                        ↓ 调用方法
                    ViewModel
                        ↓ 调 Repository
                    Repository
                    ↙        ↘
               Room(本地)    Mock/API(远程)
                    ↘        ↙
                  LiveData 返回
                    ↓
              Fragment 收到 → 自动刷新 UI
```

### 5.2 无限信息流 + 下拉刷新

**原理**：`RecyclerView.OnScrollListener` 监听滑动位置 → 滑到底部触发 `loadNextPage()`

```
PAGE_SIZE = 10, 总共12条
首次加载 → getPage(0) → [0-9]    10条
滑到底   → getPage(1) → [10-11]  2条
滑到底   → getPage(2) → []       显示"没有更多了"
```

**下拉刷新**：`SwipeRefreshLayout.setOnRefreshListener()` → `viewModel.refresh()` → `getRandomAds()` 打乱顺序

### 5.3 点赞/收藏的内外数据同步

**核心**：Room 数据库是唯一数据源 (Single Source of Truth)

```
       Room 数据库 (user_behaviors 表)
              ↙           ↘
    主界面 Adapter         详情页
    getLikeCount()        likeCount
    .observe()            .observe()
              ↘           ↙
          数据库写入了 → 自动通知所有观察者 → 一起刷新
```

**一次点赞操作的数据流**：

```
用户点 ❤
  → InteractionBar.onLikeClick()
  → DetailViewModel.toggleLike()
  → BehaviorRepository.toggleLike()
  → executor.execute() → Room INSERT 记录
  → Room 检测到数据变化 → 通知所有 LiveData 观察者
  → 主界面红心变亮 + 数字 +1
  → 详情页红心变亮 + 数字 +1
  → 同时生效，无需手动传参
```

### 5.4 AI 对话搜索

**两套实现，自动切换**：

| 情况 | 实现 | 效果 |
|------|------|------|
| 填了 API Key | Retrofit → DeepSeek API | AI 分析全部 12 条广告，直接返回匹配的 ID |
| 没填 API Key | MockAiService → 78词关键词映射表 | 本地关键词 → 标签 → 广告匹配 |

**大模型模式流程**：

```
用户输入 "手机"
  → SearchViewModel.sendMessage()
  → AiChatHelper.analyzeAndMatch()
  → 构造 Prompt:
     "你是广告推荐助手。广告库:
      [ad_001] 华为Mate... | 标签:科技,新品,热卖
      [ad_006] 小米14...  | 标签:科技,数码,热卖
      ...（全部12条）"
  → POST https://api.deepseek.com/v1/chat/completions
  → AI返回 JSON: {"reply":"为你找到2款手机!","ids":["ad_001","ad_006"]}
  → matchByIds() → [华为, 小米]
  → 聊天气泡展示
```

### 5.5 埋点统计（曝光 + 点击率）

**曝光记录**：
```java
// RecyclerView 滚动时，当前可见的卡片 ID 加入 Set
java.util.Set<String> impressedIds = new HashSet<>();
// 遍历可见 item → 不在 Set 里 → 记录曝光 → 加入 Set
adapter.behaviorRepo.recordImpression(id);
impressedIds.add(id);
```

**点击记录**：
```java
// 进入详情页 = 1次点击
behaviorRepository.recordClick(adItem.getId());
```

**点击率**：
```java
// 两个 LiveData 都收到数据后才计算
clickCount * 100f / impressionCount = 点击率%
```

### 5.6 视频播放器

**设计决策：主页不建播放器**。

| 为什么 | 解释 |
|--------|------|
| 不闪 | RecyclerView 回收 item 会销毁 SurfaceView，重建时画面黑闪 |
| 不卡 | 多个播放器实例争抢解码资源 |
| 省电 | 只有一个实际运行的 ExoPlayer |

**流程**：
```
主页:  视频卡片 = 封面图(Glide加载) + ▶ 按钮 (没有播放器)
                   ↓ 点 ▶
详情页: setupVideo() → ExoPlayer → 有声全屏
                   ← 返回 → onDetachedFromWindow → releasePlayer()
```

### 5.7 详情页 AI 分析 + 换一个

```
进入详情页
  → loadAiAnalysis()
  → API: Prompt = "你是广告分析专家。标题:华为Mate, 描述:搭载麒麟芯片..."
  → AI 返回分析文本（格式: 👥目标人群 / 💡卖点亮点 / 🎯购买建议）
  → 显示在分析卡片里
  → 用户点 [⟳ 换一个] → 重新调 API → 显示新分析 (temperature=0.7 每次不同)
```

## 六、各功能方案对比

### 6.1 信息流展示

| | 当前方案 | 选型理由 | 未来升级 |
|---|---------|---------|---------|
| 列表控件 | RecyclerView | 比 ListView 性能高，支持多 ViewType | Paging3 |
| 图片/封面 | Glide + 本地asset图片 | 零网络依赖，视觉效果真实 | 网络图片 + CDN |
| 下拉刷新 | SwipeRefreshLayout | 一行代码，Material Design 原生 | 自定义下拉动画 |
| 分页 | 手动 subList | 简单直观，学习成本低 | Paging3 自动管理加载状态 |

### 6.2 数据持久化

| | 当前方案 | 选型理由 | 未来升级 |
|---|---------|---------|---------|
| 数据库 | Room | 编译时检查 SQL，比手写 SQLite 安全 | Room + 远程同步 |
| 状态同步 | LiveData 自动推 | 生命周期安全，不内存泄漏 | Flow(协程) |
| 线程 | ExecutorService 单线程池 | 保证数据库写操作顺序执行 | Kotlin协程 |

### 6.3 视频播放

| | 当前方案 | 选型理由 | 未来升级 |
|---|---------|---------|---------|
| 引擎 | ExoPlayer (media3) | Google 官方，比 MediaPlayer 灵活 | DASH/HLS 流媒体 |
| 主页播放 | 封面图+▶按钮 | 防 RecyclerView 闪烁，点▶跳详情页有声播放 | 真机预加载 |
| 缓存 | 预留扩展点 | 架构支持 CacheDataSource | 100MB 磁盘缓存 |
| 源 | res/raw 本地 MP4 | 秒开，演示效果好 | 网络 URL + CDN |

### 6.4 AI 搜索

| | 当前方案 | 选型理由 | 未来升级 |
|---|---------|---------|---------|
| 分析方式 | AI 直接选广告 ID | 比标签匹配准 | 向量语义检索 |
| API | DeepSeek + Retrofit | OpenAI 兼容格式通用 | 多模型路由器 |
| 兜底 | Mock 78词映射表 | 没有 API 也能跑 | 增强语义理解 |
| 对话上下文 | 保留最近 3 轮 | 基本连续性 | 长会话记忆 |

## 七、已知设计缺陷与未来改进方案

> 本节诚实记录当前架构中的不足之处，以及业界成熟的解决方案。

### 7.1 Tab 切换：单一列表 + 过滤的Bug

**当前做法**：
```java
// MainFeedViewModel.switchTab()
public void switchTab(String tab) {
    if ("推荐".equals(tab)) {
        data = repository.loadInitialAds();    // 12条全部推荐
    } else {
        data = repository.loadAdsByCategory(tab, 0);  // 按分类过滤
    }
    adList.setValue(data);
}
```

**为什么会出Bug**：
- "关注"和"热门"Tab 的数据是从 12 条广告里**过滤**出来的
- 过滤后的数据太少（"关注"只有 4 条），下拉刷新、无限滚动的行为和"推荐"不一致
- 切换回"推荐"时需要重新加载全部数据，页码重置逻辑容易出错
- 三个 Tab 共用同一个 `adList` LiveData，Tab 切换时 `observer` 触发时机不可控
- Mock 数据里"关注"标签的广告实在太少，`Collections.shuffle` 对 4 条数据意义不大

**业界成熟方案**：**每个 Tab 独立 Fragment + 独立 ViewModel**

```
推荐 Tab                        关注 Tab                        热门 Tab
MainFeedFragment               FollowingFragment               HotFragment
    ↓                               ↓                               ↓
RecommendViewModel              FollowingViewModel              HotViewModel
    ↓                               ↓                               ↓
AdRepository.loadRecommend()    AdRepository.loadFollowing()    AdRepository.loadHot()
```

**优势**：
- 每个 Tab 有自己的 ViewModel 和 LiveData，互不干扰
- 每个 Tab 独立的分页状态，切回来数据还在
- 用 `ViewPager2 + TabLayout + FragmentStateAdapter` 实现，Google 官方推荐
- 滑到哪个 Tab 加载哪个，节省资源

| 对比 | 当前 (单列表过滤) | 未来 (多Fragment独立) |
|------|-----------------|---------------------|
| 数据独立性 | 共用 List，切 Tab 数据被覆盖 | 各自独立，切回来还在 |
| 分页状态 | 共用页码，切 Tab 重置 | 每个 Tab 自己的页码 |
| 下拉刷新 | 过滤列表太短，shuffle 没意义 | 每个 Tab 独立请求数据 |
| 内存 | 少 | 略多(可接受) |
| 实现复杂度 | 简单 | 中等(ViewPager2+多Fragment) |

### 7.2 AI 搜索：每次重新分析全部广告

**当前做法**：用户每问一句，把 12 条广告的标题/描述/标签全部发给 AI。

**问题**：
- 每次调 API 都传 12 条广告的全量文本，token 消耗大
- 没有缓存，相同问题重复调 API
- 广告多了（100 条+）就会超 API 的 context 长度限制
- 用户连续对话时，每次都重新分析全量广告库

**业界方案**：**向量检索 (Embedding + Vector Search)**

```
用户输入 "手机"
  ↓
用 Embedding 模型将"手机"转成 768 维向量 [0.12, -0.34, 0.56, ...]
  ↓
在本地向量库里用余弦相似度搜索 → 找到 Top3 匹配广告
  ↓
把 Top3 广告发给大模型做最终分析 → 生成推荐话术
```

**优势**：
- 不随广告数量增长而消耗更多 token
- 可以缓存广告向量，只在广告新增/修改时重建
- 语义相似度比关键词匹配精准

**当前阶段可接受的过渡方案**：
- 广告数控制在 20 条以内，每次全量发给 AI，结果准确、足够演示
- 保留 `MockAiService` 关键词匹配作为无 API 的降级方案

### 7.3 视频播放：无缓存 + RecyclerView 闪烁

**问题**：
- 每次进详情页都重新加载视频（虽然本地 MP4 很快）
- 构造时尝试过 `CacheDataSource`，但 media3 1.6.1 的 API 标记为 `@UnstableApi`，兼容性差
- RecyclerView 里放 `SurfaceView` 天然会闪烁，目前用"主页不放播放器"规避，但体验不如真正的静音自动播放

**业界方案**：

| 问题 | 升级方案 |
|------|---------|
| 无缓存 | 升级 media3 到 1.7+ 稳定版 `CacheDataSource`，100MB 磁盘缓存 |
| 主页闪烁 | 真机上 `SurfaceView` 闪烁几乎不可见（模拟器才明显）；或者换成 `TextureView` 慢慢调 |
| 主页静音播放 | `player.setVolume(0f)` + `player.play()` + 仅当前可见 item 播放、其余暂停 |
| 预加载 | 当前可见 item 的下一个提前 `prepare()`，滑到后秒播 |

### 7.4 Room 数据库：单用户设计

**当前做法**：所有用户行为（点赞/收藏/评论/曝光）直接写入，不区分用户。

**问题**：
- 表里没有 `user_id` 字段，无法区分"张三点了赞"还是"李四点了赞"
- 多用户数据混在一起，统计和推荐都不可靠

**业界方案**：
- `UserBehavior` 表加 `user_id` 字段
- 查询时带上 `WHERE user_id = :userId` 条件
- 引入账号体系 (Firebase Auth / 自建)

### 7.5 埋点统计：同质化计数器

**当前做法**：所有"曝光"和"点击"用同一个 `behavior_type` 表存储。

**问题**：
- 曝光和点击混在 `user_behaviors` 表里，和点赞/收藏/评论同一个结构
- 没有时间维度——无法区分"今天点击 3 次"和"这个月点击 3 次"
- 没有区分来源——无法知道是从"推荐"Tab 点的还是从"热门"Tab 点的

**业界方案**：

| 问题 | 升级方案 |
|------|---------|
| 数据混在一起 | 独立 `analytics_events` 表，字段: event_type / source / page / timestamp |
| 无时间分析 | 加 `created_at` 索引，SQL 按天/周/月聚合 |
| 无归因 | 加 `referrer` 字段(从哪个页面/哪个Tab来的) |
| 数据量大 | 对接埋点SDK(Firebase Analytics / 自建) + 大数据数仓离线计算 |

### 7.6 UI 方案：本地图片 + 色块占位

**当前做法**：12 张本地 PNG 图片放在 `assets/media/images/`，通过 `file:///android_asset/...` 路径用 Glide 加载。

```java
// MainFeedFragment.onBindViewHolder()
Glide.with(holder.mainImageView)
    .load(item.getImageUrl())      // "file:///android_asset/media/images/ad_tech_1.png"
    .centerCrop()
    .into(holder.mainImageView);
```

**为什么用本地图片**：
- 零网络依赖，秒加载，演示效果稳定
- 色块作为加载过程中的占位背景，图片加载完成后覆盖
- 未来接 CDN 图片只需换 `imageUrl` 字段，代码不用改

**色块角色变化**：
- 旧方案：色块是唯一视觉元素（马卡龙色区分分类）
- 新方案：色块只是 Glide 加载过程中的占位背景，加载完成后被图片覆盖
- 信息流卡片隐藏了色块叠加文字（`bannerText.setVisibility(GONE)`），详情页也同步隐藏

> 架构上已在 `build.gradle` 引入 Glide 依赖，当前已切换为真实图片方案。

## 九、答辩要点全覆盖

### 9.1 项目背景
> "训练营课题是实现AI广告推荐信息流App，个人参赛。作为Android零基础学员，从Activity生命周期、五种基础布局开始学，到最终交付了一个功能完整的应用。"

### 9.2 架构设计
> "采用 Google 推荐的 MVVM + Repository 架构：View 层只做展示和事件转发，ViewModel 管理所有 UI 状态并通过 LiveData 驱动更新，Repository 作为数据门面隐藏来源细节。引入面向接口编程的 AI 服务层，Mock 和真实 API 实现可互换。"

### 9.3 数据同步机制
> "点赞/收藏状态通过 Room 数据库的 LiveData 机制实现内外同步。主界面 Adapter 和详情页都通过 `getLikeCount(adId).observe()` 观察同一条数据库查询——数据写入 Room 后自动推送给所有观察者，无需 EventBus 或手动回调。这是 MVVM 架构中'单一数据源'思想的体现。"

### 9.4 视频播放方案
> "ExoPlayer 封装为 VideoPlayerView 组件。主页列表显示静态色块+播放按钮，点击跳详情页才初始化播放器——这是为了避免 RecyclerView 中 SurfaceView 销毁重建造成的闪烁问题。`onDetachedFromWindow` 自动释放资源防止内存泄漏。架构上预留了 CacheDataSource 扩展点供后续上线使用。"

### 9.5 AI 搜索实现
> "对话式广告推荐采用大模型直接分析本地广告库的方案。将12条广告的标题、标签、描述打包发给大模型，AI 直接返回匹配的广告ID和推荐理由。相比传统'标签→匹配'方案更精准——比如用户说'手机'，AI 会区分广告的语义并选出华为和小米。同时设计了 Mock 兜底：未配置 API Key 时自动降级为本地关键词匹配，保证功能完整性。"

### 9.6 组件化与复用
> "InteractionBar 和 VideoPlayerView 是两个可复用组件。InteractionBar 在主界面的大图卡片、小图卡片和详情页三处复用——通过 `setLiked()` 和 `setOnInteractionListener()` 暴露简单接口，内部封装了图标切换、数字格式化、心形弹跳动画。VideoPlayerView 封装了 ExoPlayer 的完整生命周期：对外只暴露 `setupVideo/play/pause/setMuted` 四个方法，对内管理播放器的创建、准备、释放全流程。"

### 9.7 学习能力体现
> "从Android零基础开始，两周内完成了：①五类核心控件的使用(RecyclerView/ViewPager2/TabLayout/自定义View)；②Room数据库的完整CRUD及LiveData同步；③Retrofit网络请求接入大模型API；④ExoPlayer视频播放及生命周期管理；⑤MVVM架构理解和落地。项目从架构设计到组件封装到功能实现均为独立完成。同时预留了Glide图片加载/CacheDataSource视频缓存/Paging3分页等升级扩展点，体现了对生产级应用架构的理解。"

### 9.8 踩坑与解决
> "①主页视频闪烁：SurfaceView在RecyclerView中回收重建导致画面黑闪 → 改为静态封面+按钮方案；②详情页崩溃：ViewModel有参构造需要自定义Factory，忘了传导致ClassCastException → 统一用ViewModelProvider.Factory；③ANR卡死：连续两次LiveData.setValue触发两次RecyclerView全部刷新 → 合并为一次批量更新；④JSON解析失败：description里的中文双引号被Java当成字符串结束符 → 统一用「」替代。"

### 9.9 未来上线方案
> "当前基于 Mock 数据的 MVP 版本验证了核心链路。上线时：数据源切换为推荐引擎 API，图片加载接入 Glide + CDN，视频播放加 CacheDataSource 边播边缓存，AI 搜索接入更强的模型并加向量检索，统计模块对接埋点 SDK，架构层面引入 Hilt 依赖注入和 Clean Architecture 分层，补充单元测试和 CI/CD 流水线。"
